## 安全设计原则

Asterism 从设计层面遵循以下安全原则：

1. **沙箱运行** — `.aster` 脚本不具备文件系统写入、网络访问、外部进程启动能力。它只能操作游戏内状态（变量、旗标、场景切换）。
2. **输入校验** — 所有外部输入（脚本内容、用户按键、资源文件）必须经过合法性校验。
3. **资源完整性** — 资源文件加载前检查魔数（magic bytes），存档文件校验 CRC32 完整性。
4. **零硬编码密钥** — 代码中不包含任何硬编码的密钥、密码或 Token。
5. **依赖审计** — CI 集成 `cargo audit` 和 `pnpm audit`，存在已知严重漏洞的依赖将被阻止合并。
6. **最小权限** — WASM 插件（v1.0.0）基于 Wasmtime 沙箱运行，默认无文件系统/网络/线程权限。

## 报告安全漏洞

### 我们不希望看到的

> ⚠️ **请勿**在公开的 GitHub Issue 中报告安全漏洞。这会使漏洞在修复前被公开利用。

### 正确的报告方式

1. **发送邮件**至 `security@asterism.dev`（待项目正式上线后启用）
2. 在当前阶段，请通过 GitHub 的 **[私有漏洞报告（Private Vulnerability Reporting）](https://github.com/asterism-engine/asterism/security/advisories/new)** 功能提交
3. 报告应包含：
   - 漏洞的详细描述
   - 复现步骤（含最小化 `.aster` 脚本或项目，如适用）
   - 受影响的版本和平台
   - 潜在影响评估
   - 可能的修复建议（如你有想法）

### 我们承诺

- ⏰ **48 小时内**确认收到报告
- 📅 **7 天内**提供初步评估结果
- 🔧 **30 天内**发布修复（严重漏洞将更快）
- 📢 修复后发布安全公告（CVE 申请视严重程度决定）
- 🏆 在安全公告中致谢报告者（除非你要求匿名）

## 依赖安全

Asterism 使用以下工具持续监控依赖安全：

| 工具 | 用途 | 频率 |
|------|------|------|
| `cargo audit` | Rust crate 漏洞扫描 | 每 PR / 每日 |
| `pnpm audit` | npm 包漏洞扫描 | 每 PR / 每日 |
| `cargo deny` | 许可证合规检查 | 每 PR |
| Dependabot | 自动依赖更新 PR | 实时 |

依赖更新策略：
- **安全更新**：严重/高危漏洞 → 立即修复
- **patch 更新**：自动合并（CI 通过即可）
- **minor 更新**：Review 后合并
- **major 更新**：需项目维护者手动评估

## 安全最佳实践（贡献者指南）

如果你正在为 Asterism 贡献代码，请遵守以下安全实践：

1. **永远不要信任输入** — 对所有外部输入（文件内容、用户操作、网络数据）进行校验
2. **使用 `?` 而非 `unwrap()`** — 不当的错误处理可能导致 panic 或未定义行为
3. **不要硬编码敏感信息** — 密钥、密码、Token 等一律通过配置文件注入
4. **最小权限原则** — 每个模块只能访问它需要的资源
5. **安全 CHANGELOG** — 如果你的 PR 涉及安全相关变更，请在描述中明确标注

### Rust 安全检查清单

```rust
// ✅ 正确：对输入进行边界检查
fn load_save_slot(slot_index: usize) -> Result<SaveData, SaveError> {
    if slot_index >= MAX_SAVE_SLOTS {
        return Err(SaveError::SlotOutOfRange { index: slot_index });
    }
    // ...
}

// ❌ 错误：信任外部输入长度
fn load_save_slot(slot_index: usize) -> SaveData {
    let slot = &self.slots[slot_index]; // 可能 panic！
    // ...
}
```
